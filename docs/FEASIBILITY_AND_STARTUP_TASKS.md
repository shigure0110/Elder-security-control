# 可行性评估 + 实现建议 + 启动任务

> 针对你的 3 个核心需求，先明确“能不能做、怎么做、做到什么程度”。

## 1) 每日在微信群汇报今日开销（支付宝 + 微信）

### 可行性结论
- **可以做**“稳定可用版本”，但需要接受平台能力边界：
  - 支付宝/微信通常没有对普通第三方 APK 开放稳定的账单读取 API。
  - 因此建议采用多源融合：**通知读取 + 辅助功能识别 + 用户导出账单兜底**。

### 推荐实现路线
1. **账单采集层**
   - `NotificationListenerService` 采集支付通知（金额、商户、时间）。
   - Accessibility 仅在用户授权下读取关键页面文本（高风险动作识别/补全字段）。
   - 支持手工导入 CSV/账单文件作为补偿。
2. **账单标准化层**
   - 标准字段：平台、订单号、金额(分)、商户、时间、类别、来源可信度。
   - 去重规则：平台 + 订单号 + 时间窗。
3. **日报生成层**
   - 以“小额阈值”筛选（默认 ≤ 200 元）。
   - 输出：总笔数、总金额、Top 商户、异常摘要。
4. **微信群投递层**
   - 目标群名称 + 群唯一 ID 配置。
   - 投递失败重试 + 状态记录。

## 2) 新安装软件、微信加好友需要家庭同意，否则阻止

### 可行性结论
- **“绝对阻止安装”**：通常需要 Device Owner / 企业管控模式；普通民用设备很难 100% 底层拦截。
- **“微信加好友绝对阻止”**：无官方拦截 API；可行路径是“检测到动作后覆盖锁 + 远程审批解锁”。

### 推荐实现路线
1. **检测层**
   - 安装行为：监听安装相关广播/系统事件 + 应用列表变化。
   - 微信加好友：Accessibility 检测关键页面/控件语义。
2. **策略层**
   - 规则：高风险动作默认锁定。
   - 发起审批：向家属端推送“同意/拒绝”请求。
3. **执行层**
   - 普通模式：覆盖拦截界面 + 倒计时 + 远程放行 token。
   - Device Owner 模式：可接入更强限制能力（按机型与 ROM 验证）。
4. **审计层**
   - 记录每次动作、审批人、审批结果、放行时长。

## 3) 启动任务（可直接开工）

### Sprint 1（1~2 周）
- [ ] 完成设置中心字段落库（本地 + 服务端同步）
- [ ] 接入支付宝/微信通知采集与账单标准化
- [ ] 生成“每日小额账单摘要”
- [ ] 支持固定微信群路由投递（单群）

### Sprint 2（2 周）
- [ ] 新安装应用检测 + 家庭审批链路（提醒/拦截）
- [ ] 微信加好友页面检测 + 审批放行 token
- [ ] 失败重试、投递追踪、审计日志

### Sprint 3（2~4 周）
- [ ] 多平台扩展（抖音/拼多多/淘宝）
- [ ] 多群路由（日报群 / 告警群）
- [ ] 家属端趋势报表与原因解释
